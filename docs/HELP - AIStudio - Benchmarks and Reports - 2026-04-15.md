# Understanding Benchmark and Report Files

A guide to the benchmark outputs AIStudio generates, what they mean, and how to read them.

---

## Overview

AIStudio produces two types of output files you'll encounter in the repo:

| Type | Location | Generated by |
|------|----------|--------------|
| Benchmark reports | `benchmarks/reports/benchmark_<corpus>_<timestamp>.md` | `benchmarks/benchmark.py` |
| Demo run reports | `data/<corpus>/reports/report_<timestamp>_<model>.md` | `scripts/run_demo.py` (legacy) |

The benchmark harness (`benchmarks/benchmark.py`) is the current tool. Demo reports are from earlier development and are kept for historical reference.

---

## How to Run a Benchmark

```bash
cd ~/Developer/AIStudio && source .venv/bin/activate
python3 benchmarks/benchmark.py --corpus demo --top-k 5 --temperature 0.3
```

Options:
- `--corpus` — which corpus to query (must be ingested)
- `--top-k` — chunks retrieved per query (default: 5)
- `--temperature` — LLM temperature (default: 0.3)
- `--model` — override default model (e.g., `--model llama3.1:70b`)

Questions are read from `benchmarks/{corpus}_questions.yaml`. Output written to `benchmarks/reports/`.

## Reading a Benchmark Report

Each report has this structure:

### Header block
Configuration used for the run: corpus, top-k, temperature, model, API endpoint. Use this to compare runs — two reports are only comparable if the configuration is identical.

### Summary block
- **Questions** — total questions in the question file
- **Passed** — how many passed the automated evaluator
- **Avg latency** — mean response time across all questions

**Important caveat on pass/fail:** The automated evaluator requires both correct content AND an inline `[N]` citation marker. A factually correct answer with no citation marker counts as a fail. This means the automated pass rate understates actual retrieval quality. Always read the individual results to distinguish "wrong answer" failures from "correct but uncited" failures.

### Infrastructure block
Snapshot of the stack at run time: Qdrant version, embedding model, corpus size, ingest throughput. Useful for comparing runs across machines or after stack changes.

### Results table
One row per question:
- **Latency** — wall-clock time for that query
- **Pass** — ✅ passed evaluator | ❌ failed
- **Citations** — sources returned
- **Notes** — what the answer drew from, any retrieval observations

### Detailed Results section
Full question text, filter settings used, latency, pass/fail, and the actual answer returned by the model (truncated). Use this to diagnose individual failures — you can see exactly what the model said and where retrieval went wrong.

### Known Limitations section
Issues observed during this run. Common ones:
- **Citation compliance** — model answers correctly but omits `[N]` markers. More common on 8b than 70b.
- **Vocabulary mismatch** — query phrasing and source text use different terms for the same concept. Fix: CrossEncoder reranker (Wave 2).
- **XBRL noise** — structured data tags in SEC 10-K HTML filings create low-quality chunks.

### Roadmap section
Items identified during benchmarking that would improve results.

## The Warm Model Insight (Key Benchmark Finding)

From Phase 1 benchmarking on Apple Silicon M4:

> Once a model is loaded into unified memory, `llama3.1:70b` and `llama3.1:8b` respond at statistically identical latency — ~6–7s average. Model size stops being a latency variable after cold start.

Practical implication: use 70b for quality-sensitive work. The latency cost vs. 8b is zero once warm. AIStudio's prewarm endpoint (`POST /prewarm`) loads the model into memory on startup so the cold-start penalty is invisible to the user.

---

## Comparing Runs

To compare two benchmark reports meaningfully, they must share:
- Same corpus and question file
- Same top-k and temperature
- Same model (or explicitly compare model variants)

Latency comparisons are only valid warm-vs-warm. A cold run will show 10–20× higher latency than a warm run — this is expected, not a regression.

## Demo Reports (Legacy)

Files under `data/<corpus>/reports/report_<timestamp>_<model>.md` were generated by the earlier `run_demo.py` harness. They follow a similar structure but reference ChromaDB (the previous vector store, replaced by Qdrant) and a JSONL lexical search layer (removed). Treat these as historical artifacts — do not use them to evaluate current system performance.

Current stack: **Qdrant only** (no ChromaDB, no JSONL lexical search).
